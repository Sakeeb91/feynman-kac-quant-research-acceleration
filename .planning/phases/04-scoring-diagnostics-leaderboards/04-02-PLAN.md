  <action>
    **Async orchestrator (`async_orchestrator.py`):**

    1. Add a `scoring_config: ScoringConfig | None = None` parameter to `run_batch_async()` and `resume_batch_async()`. Default to `ScoringConfig()` if None.
    2. Import `get_scorer` from `scoring.registry`, `diagnose_convergence` from `diagnostics.health`, and `assign_pareto_scores` from `scoring.pareto`.
    3. In `run_batch_async()`: Resolve the scorer at the top: `scorer = get_scorer(scoring_config)`. Pass `scorer` down through `_execute_scenarios_concurrent()` and `_execute_scenario_safe()` to `_submit_and_poll_scenario()`.
    4. In `_submit_and_poll_scenario()`: Replace `record["score"] = compute_score(record)` with `record["score"] = scorer(record)`. After building the record, add `record["convergence_health"] = diagnose_convergence(record).value`.
    5. In `_build_failure_record()`: Add `"convergence_health": "exploding"` to the failure record dict (failed scenarios are always exploding per research decision).
    6. After `_execute_scenarios_concurrent()` returns in `run_batch_async()`: If `scoring_config.strategy == ScoringStrategy.PARETO_MULTI_OBJECTIVE`, call `assign_pareto_scores(records, objectives=scoring_config.pareto_objectives)` and update each record's `score` with the Pareto-assigned score. Then re-sort by score.
    7. Thread `scoring_config` from CLI through `run_batch_async()` and `resume_batch_async()`.

    **Sync orchestrator (`orchestrator.py`):**

    8. Add the same `scoring_config: ScoringConfig | None = None` parameter to `run_batch()`.
    9. Resolve `scorer = get_scorer(scoring_config)` at the top of `run_batch()`.
    10. Replace `record["score"] = compute_score(record)` with `record["score"] = scorer(record)`. Add `record["convergence_health"] = diagnose_convergence(record).value` after building each record.
    11. In `_build_failure_record()` (sync version): Add `"convergence_health": "exploding"`.
    12. After the polling loop: If Pareto strategy, apply `assign_pareto_scores()` post-hoc.

    **IMPORTANT threading details:**
    - The `scorer` callable must be passed through the function chain. Add a `scorer` parameter to `_execute_scenarios_concurrent()`, `_execute_scenario_safe()`, and `_submit_and_poll_scenario()`.
    - Remove the import of `compute_score` from `reporting` in both orchestrators (it's now only needed as a backward-compat wrapper in reporting.py itself).
    - For `resume_batch_async()`: Since it doesn't have access to the manifest/ScoringConfig, use the default `ScoringConfig()` (loss_based with default weight). A future enhancement could store the ScoringConfig in the DB.

    **Tests:**

    In `tests/test_orchestrator.py` (add alongside existing tests):
    - `test_run_batch_records_have_convergence_health`: Mock the client, run a batch, verify every returned record has a `convergence_health` key.
    - `test_run_batch_failure_record_has_convergence_health`: Force a scenario failure, verify the failure record has `convergence_health: "exploding"`.

    In `tests/test_async_orchestrator.py` (add alongside existing tests):
    - `test_run_batch_async_records_have_convergence_health`: Same as sync test but for the async path.
    - `test_run_batch_async_failure_record_convergence_health`: Same for failure.
    - `test_run_batch_async_pareto_rescoring`: Pass `scoring_config=ScoringConfig(strategy=ScoringStrategy.PARETO_MULTI_OBJECTIVE)`. Verify that after execution, records are sorted by Pareto-derived scores (not just raw train_loss).

    **Backward compatibility:** Existing tests that don't pass `scoring_config` must still work unchanged because the default `ScoringConfig()` replicates the old `compute_score()` behavior exactly.
  </action>
