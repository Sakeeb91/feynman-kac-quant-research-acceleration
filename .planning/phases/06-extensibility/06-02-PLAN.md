**1. DB Schema v3 -> v4 migration (`store/migrations.py`):**
- Bump `CURRENT_SCHEMA_VERSION` from 3 to 4
- Add `_migrate_v3_to_v4(conn)`: `ALTER TABLE batch_runs ADD COLUMN problem_id TEXT DEFAULT 'black_scholes'`
- Register in migrations dict: `3: _migrate_v3_to_v4`

**2. MetadataStore problem_id support (`store/metadata.py`):**
- Add `problem_id: str = "black_scholes"` parameter to `create_batch_run()`
- Add `problem_id` to the INSERT statement and parameter tuple
- Existing callers that don't pass `problem_id` get the default `"black_scholes"` -- backward compatible

**3. Preflight validation with ProblemSpec delegation (`validation/preflight.py`):**
- Import `get_problem_spec` from `fk_quant_research_accel.problems`
- Add a new step at the TOP of `validate_manifest()`: resolve the ProblemSpec from `manifest.problem_id` via `get_problem_spec()`. If `problem_id` is invalid, catch `ValueError` and append a `PreflightError(field="problem_id", value=manifest.problem_id, message=str(exc))`. Return early with this error (no point validating further if problem type is unknown).
- Add scoring strategy validation: call `problem_spec.supports_scoring_strategy(manifest.scoring.strategy.value)`. If False, append `PreflightError(field="scoring.strategy", value=manifest.scoring.strategy.value, message=f"Problem type '{manifest.problem_id}' does not support scoring strategy '{manifest.scoring.strategy.value}'")`.
- Keep existing Black-Scholes-specific validation as-is for now -- it runs unconditionally. In a future refactor, this can be moved into BlackScholesSpec.validate(). For Phase 6, the key change is: (a) unknown problem_id is caught, (b) scoring strategy compatibility is checked.
- IMPORTANT: Do NOT break existing preflight behavior. The existing BS-specific checks (volatility, correlation, dimension-option) must still run when problem_id is "black_scholes".

**4. Problem-agnostic comparison key (`run_analysis/comparison.py`):**
- Replace the hardcoded `_scenario_key()` that extracts dim/volatility/correlation/option_type/model_config with a deterministic hash of the entire `scenario_json` string.
- New implementation: `_scenario_key(scenario_json: str) -> str` -- parse JSON, re-serialize with `json.dumps(sorted_keys=True)` to normalize, return the normalized string as key. This works for any problem type because it uses the full scenario payload.
- Update the return type annotation from `tuple[Any, ...]` to `str`.

**5. Update tests:**
- `tests/test_store.py`: Add test for v3->v4 migration (verify `problem_id` column exists after migration, verify default is `"black_scholes"`). Add test that `create_batch_run` with explicit `problem_id="harmonic_oscillator"` persists correctly.
- `tests/test_validation.py`: Add test that `validate_manifest` with invalid `problem_id` (e.g., "nonexistent_problem") returns a PreflightError for field "problem_id" listing valid IDs. Add test that scoring strategy rejection works (if we make HO reject some strategy, test it).
- `tests/test_run_analysis.py`: Verify `_scenario_key` still aligns scenarios correctly after the change to deterministic JSON key. Existing comparison tests should pass if the alignment logic is correct.
