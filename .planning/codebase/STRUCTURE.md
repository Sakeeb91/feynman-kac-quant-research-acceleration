## Naming Conventions

**Files:**
- Python source: `snake_case.py` (e.g., `client.py`, `orchestrator.py`, `reporting.py`)
- Test files: `test_<module>.py` (e.g., `test_reporting.py`)
- Config files: Descriptive lowercase (e.g., `pyproject.toml`, `README.md`)

**Directories:**
- Package directory: `snake_case` (e.g., `fk_quant_research_accel`)
- Top-level folders: lowercase (e.g., `src`, `tests`, `.planning`)

**Classes:**
- PascalCase (e.g., `FKPinnClient`, `BatchConfig`, `Scenario`)
- Dataclasses marked with `@dataclass(frozen=True)` for immutability

**Functions:**
- snake_case (e.g., `generate_black_scholes_scenarios`, `run_batch`, `write_csv`, `compute_score`)
- Private methods prefixed with `_` (e.g., `_get`, `_post`, `_url`, `_parse_int_list`)

**Variables/Constants:**
- snake_case for local/instance variables
- ALL_CAPS for module-level constants (e.g., `TERMINAL_STATUSES`)

## Where to Add New Code

**New Feature (e.g., additional problem type):**
- Primary code: Add new generator function in `src/fk_quant_research_accel/orchestrator.py` (pattern: `generate_<problem>_scenarios()`)
- Config updates: Extend `BatchConfig` or create specialized config class
- CLI updates: Add subcommand or new argument group in `src/fk_quant_research_accel/cli.py`
- Tests: Add test cases to `tests/test_reporting.py` or create `tests/test_orchestrator.py`

**New Utility Function (e.g., filtering, validation):**
- Reporting utilities: `src/fk_quant_research_accel/reporting.py`
- Client utilities: `src/fk_quant_research_accel/client.py` (if HTTP-related)
- Orchestrator utilities: `src/fk_quant_research_accel/orchestrator.py` (if scenario-related)

**New Scoring Method:**
- Location: `src/fk_quant_research_accel/reporting.py` (add alongside `compute_score()`)
- Pattern: Pure function accepting record dict, returning float score
- Update: Modify `run_batch()` in orchestrator.py to use new scorer if desired

**Testing:**
- Location: `tests/` directory
- File: Create `tests/test_<module>.py` matching the module being tested
- Pattern: Use pytest fixtures for common test data

## Project Structure Conventions

**Import organization:**
- Absolute imports from package root (e.g., `from fk_quant_research_accel.client import FKPinnClient`)
- Standard library imports first, then third-party, then local (observed in cli.py, orchestrator.py)
- No relative imports (e.g., `from .client import FKPinnClient` used in __init__.py only for exports)

**Module organization:**
- One major class/abstraction per module
- Related helper functions in same module
- CLI as separate module (thin wrapper around orchestrator/reporting APIs)

**Configuration:**
- All config via `pyproject.toml` (setuptools, ruff, pytest)
- Runtime args via CLI flags (no config files)
- Inline defaults in function signatures (e.g., `poll_seconds=1.5` in client.py)
